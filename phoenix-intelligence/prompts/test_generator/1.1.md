## Locator Priority (follow strictly)
1. `get_by_role` — default for all interactive elements. Always provide `name=`.
2. `get_by_label` — form fields with visible labels.
3. `get_by_placeholder` — inputs without visible labels.
4. `get_by_text` — static text content ONLY when the text is unique on the page. NEVER use in assertions.
5. `get_by_test_id` — when a `data-testid` attribute exists.
6. CSS selector — only when semantic locators are not possible.
7. XPath — absolute last resort.

## ⛔ Strict-Mode Anti-Patterns (these cause test failures — never do them)

### 1. get_by_text() in assertions
**WRONG** — get_by_text() matches ALL elements containing the text. If it appears in a nav link AND a heading, Playwright raises "resolved to 2 elements".
```python
# WRONG
expect(page.get_by_text("Dashboard")).to_be_visible()
```
**CORRECT** — always specify the role:
```python
# Page headings / module titles
expect(page.get_by_role("heading", name="Dashboard")).to_be_visible()
# Toast / success messages
expect(page.get_by_role("alert")).to_have_text("Successfully Saved")
# Status text inside a specific container
expect(page.locator(".oxd-topbar-header-breadcrumb").get_by_text("Dashboard")).to_be_visible()
```

### 2. Hardcoded user display names in locators
**WRONG** — the logged-in user's display name ("Paul Collings", "Admin User") is dynamic and changes per environment/account.
```python
# WRONG
page.get_by_role("button", name="Paul Collings").click()
```
**CORRECT** — target the structural element, not its runtime content:
```python
# Target the user-profile dropdown in the top navigation banner
page.get_by_role("banner").get_by_role("button").last.click()
# Or use a stable CSS class if you know the framework
page.locator(".oxd-userdropdown-tab").click()   # OrangeHRM
page.locator("[data-testid='user-menu']").click() # apps with test IDs
```

### 3. get_by_text().click() without scoping
**WRONG** — same element-collision risk for clicks:
```python
# WRONG — "Dashboard" appears in nav AND in the breadcrumb
page.get_by_text("Dashboard").click()
```
**CORRECT** — scope to the container or use role:
```python
page.get_by_role("navigation").get_by_role("link", name="Dashboard").click()
# or
page.get_by_role("link", name="Dashboard").first.click()
```

### 4. Strict-mode collision rule
If two option labels share a substring (e.g. `Male` vs `Female`, `Enable` vs `Enabled`), you MUST use `exact=True`:
```python
page.get_by_label("Gender").select_option("Male")       # WRONG — also matches Female
page.get_by_text("Male", exact=True).check()            # CORRECT
```

### 5. Position-based locators
**WRONG** — `.nth(3)` or `:nth-child(3)` break when page layout changes.
```python
page.locator("tr").nth(3)  # WRONG
```
**CORRECT** — filter by content:
```python
page.get_by_role("row").filter(has_text="John Smith")  # CORRECT
```

## Waiting Rules
- NEVER use `time.sleep()` or `page.wait_for_timeout()` to fix flaky tests.
- Rely on Playwright auto-waiting for actions (.click, .fill, etc.).
- Use `page.wait_for_load_state('networkidle')` only after full-page navigations.
- Use `expect()` assertions for waiting on state changes.

## Dialog Handling
- NEVER use `page.on("dialog", ...)` — it stacks handlers across multiple dialogs.
- ALWAYS use `page.once("dialog", ...)` immediately before the action that triggers the dialog:
```python
page.once("dialog", lambda d: d.accept())
page.get_by_role("button", name="OK").click()
```

## Assertions
- Use `expect(locator)` assertions from `playwright.sync_api`.
- For URL matching use `expect(page).to_have_url(re.compile(r"..."))`.
- Never use bare `assert` with `.text_content()` — use `expect().to_have_text()`.
- Do NOT assert on imagined success messages unless the page snapshot explicitly shows that exact text.
- For error messages use `get_by_role("alert")` not `get_by_text()`.

## Naming
- Function name: `test_<short_descriptive_name>` using snake_case (max 50 chars).

## Test Structure
- Arrange: navigate and set up state.
- Act: perform the user actions described in the acceptance criteria.
- Assert: verify expected outcomes with `expect()`.
