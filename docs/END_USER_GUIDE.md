# Phoenix SmartAutomation — End User Guide

> For teams who have received `phoenix-intelligence.exe` from their administrator.
> No source code or build tools required.

---

## How Phoenix Works

```
You write a user story
        │
        ▼  phoenix generate
Phoenix reads the story → calls Claude AI → writes human-readable test cases
        │
        ▼  You review manual_tests/*.md   ← the only manual step
        │
        ▼  phoenix automate
Phoenix generates Page Objects (pages/) and test scripts (tests/)
        │
        ▼  phoenix run
Phoenix runs all tests, retries failures automatically, saves an HTML report
```

## Part 1 — Start the Intelligence Server

The server must be running before you use any `phoenix` commands. Start it once and leave it running.

### Step 1.1 — Place the executable

```
C:\Phoenix\phoenix-intelligence.exe
```

### Step 1.2 — Set your API key

```powershell
$env:ANTHROPIC_API_KEY = "sk-ant-..."
```

To set it permanently (survives reboots):
```powershell
[System.Environment]::SetEnvironmentVariable("ANTHROPIC_API_KEY", "sk-ant-...", "User")
```

### Step 1.3 — Start the server

Run from a terminal — do **not** double-click in Explorer.
```powershell
C:\Phoenix\phoenix-intelligence.exe
```

> **Windows SmartScreen alert:** First run only — click **"More info"** then **"Run anyway"**.

You will see:
```
INFO:     Application startup complete.
INFO:     Uvicorn running on http://0.0.0.0:8001
```

**Keep this terminal open.** The server must stay running.

### Step 1.4 — Verify the server

```powershell
curl http://localhost:8001/health
```

Expected: `{"status": "ok", "llm": {"configured": true, "provider": "anthropic"}}`

## Part 2 — Install Phoenix CLI

### First-time setup

```powershell
# Create and activate a virtual environment
python -m venv phoenix-venv
phoenix-venv\Scripts\activate

# Install from the wheel files (run from the folder containing .whl files)
pip install phoenix_shared-0.1.5-py3-none-any.whl
pip install phoenix_core-0.1.5-py3-none-any.whl

# Install the Playwright browser
playwright install chromium

# Verify
phoenix --version
```

### Upgrade (existing users)

```powershell
# 1. Activate your virtual environment
phoenix-venv\Scripts\activate

# 2. Uninstall old versions
pip uninstall phoenix-core phoenix-shared -y

# 3. Install new versions
pip install phoenix_shared-0.1.5-py3-none-any.whl
pip install phoenix_core-0.1.5-py3-none-any.whl

# 4. Replace the server executable (delete old, copy new to C:\Phoenix\)

# 5. Verify
phoenix doctor
```

> Your project files (`manual_tests/`, `tests/`, `.phoenixrc`) are not affected by upgrade steps.

## Part 3 — Set Up a Project

```powershell
mkdir my-project
cd my-project
phoenix init --base-url "https://your-app.com"
```

Creates:

```
my-project/
├── .phoenixrc              ← project config (edit this first)
├── conftest.py             ← Playwright fixtures
├── user_stories/           ← write your user stories here
├── domain_knowledge/       ← project-wide context (URLs, UI patterns, data rules)
├── pages/                  ← Page Object classes (generated by phoenix automate)
│   └── base_page.py        ← base class all page objects inherit from
├── manual_tests/           ← Phoenix writes test specs here
├── tests/                  ← Phoenix writes Playwright test scripts here
├── locators/               ← Phoenix writes element locators here
├── reports/                ← HTML reports
└── logs/                   ← execution logs
```

**Configure `.phoenixrc`:**
```toml
[project]
name            = "my-project"
base_url        = "https://your-app.com"
default_browser = "chromium"
layout          = "pom-v1"

[intelligence]
base_url = "http://localhost:8001/api/v1"
timeout  = 300
```

**Set your test credentials** — copy `.env` to `.env.local` and fill in:
```
APP_URL=https://your-app.com
TEST_USERNAME=your_test_username
TEST_PASSWORD=your_test_password
ANTHROPIC_API_KEY=sk-ant-...
```

Load them each session:
```powershell
Get-Content .env.local | ForEach-Object {
    if ($_ -match '^\s*([^#][^=]*)=(.*)$') {
        [System.Environment]::SetEnvironmentVariable($matches[1].Trim(), $matches[2].Trim(), "Process")
    }
}
```

**Check everything:**
```powershell
phoenix doctor
```

## Part 5 — Commands Reference

### `phoenix generate` — Create test cases from a user story

```powershell
phoenix generate --story-file user_stories/login.txt
```

Writes test cases to `manual_tests/` as Markdown files. **Open them and review** — add missing steps, fix expected results. This is the only step that requires your attention.

| Flag | What it does |
|---|---|
| `--story-file <path>` | Read the user story from a file |
| `--jira PROJ-123` | Fetch story + attachments from Jira |
| `--type both` | Generate manual + automation scripts in one step |
| `--docs <path>` | Attach a supporting document file or folder |
| `--no-gate` | Save all tests even if the quality check flags them |

> **`Generated 0 manual test(s)`?** Add `--no-gate` to save them anyway so you can edit manually.

---

### `phoenix automate` — Turn manual tests into Playwright scripts

```powershell
phoenix automate
```

With `layout = "pom-v1"` in `.phoenixrc`, produces:
- `pages/<module>_page.py` — Page Object class with all interactions
- `tests/<module>/test_<name>.py` — thin test wrapper that calls the page object
- `locators/<module>.json` — element locators with fallback strategies

| Flag | What it does |
|---|---|
| `--manual-dir <path>` | Use a different directory instead of `manual_tests/` |
| `--file <path>` | Automate a single manual test file |

**Supported filename patterns in `manual_tests/`:**
`manual_test_*.md`, `test_*.md`, `TC-*.md`, `*_manual.md`, `*_test.md`

---

### `phoenix run` — Execute the tests

```powershell
phoenix run tests/<path>
```

If a test fails, Phoenix retries it automatically (up to 3 times) and attempts a locator heal between each retry. An HTML report is saved to `reports/` after every run.

| Flag | What it does |
|---|---|
| `--headed` | Open a visible browser window — useful for debugging |
| `--slow-mo 500` | Slow down each action by N ms — use with `--headed` |
| `--failed-only` | Re-run only tests that failed in the previous run |
| `--browser firefox` | `chromium` (default), `firefox`, or `webkit` |
| `--max-attempts 5` | Increase healing retries |

---

### `phoenix fix` — Auto-repair failing scripts

```powershell
phoenix fix
phoenix run --failed-only
```

| Flag | What it does |
|---|---|
| `--dry-run` | Show what would be fixed without writing any files |
| `--run-id <id>` | Fix failures from a specific run |

---

### `phoenix clean` — Remove generated files

```powershell
phoenix clean           # delete test scripts, reports, locators, logs
phoenix clean --dry-run # preview what would be deleted
```

> **Note:** `pages/` (Page Object files) is **not** cleaned — page objects are long-lived shared code. Delete them manually if you want to regenerate from scratch.

---

### Other commands

```powershell
phoenix report --open           # generate and open HTML report in browser
phoenix report --trend          # multi-run trend report
phoenix logs                    # last 10 runs
phoenix logs --run-id <id>      # per-test detail for one run
phoenix locators                # inspect all element locators
phoenix doctor                  # check API key, server, Playwright, plugins
phoenix migrate                 # add missing dirs/files to an existing project
```

## Part 6 — Daily Workflow

```powershell
# 1. Write a user story
notepad user_stories\login.txt

# 2. Generate manual test cases
phoenix generate --story-file user_stories/login.txt

# 3. Review manual_tests/  ← only manual step

# 4. Generate Playwright scripts
phoenix automate

# 5. Run the tests
phoenix run tests/

# 6. Fix any failures
phoenix fix
phoenix run --failed-only

# 7. View the report
phoenix report --open
```

## Part 8 — Writing Good User Stories

```
User Story: Login to My App
Application URL: https://your-app.com/login

As a registered user I want to log in.

Acceptance Criteria:

Scenario 1: Successful Login
- Navigate to the login page
- Enter Username 'standard_user' and Password 'secret_sauce', then click Login
- Verify the Products page is displayed

Scenario 2: Invalid Password
- Enter a correct username and wrong password, then click Login
- Verify an error message is shown

Scenario 3: Empty Fields
- Leave Username and Password empty and click Login
- Verify a validation error is shown
```

**Tips:**
- One scenario per logical flow (happy path, error, edge case)
- Name the fields users fill in and what they should see after each action
- Plain English — no technical jargon needed

## Part 9 — Troubleshooting

### `phoenix` command not found
Your virtual environment is not active:
```powershell
phoenix-venv\Scripts\activate
```

---

### `Generated 0 manual test(s)`
The quality gate rejected the generated tests. Add `--no-gate` to save them anyway:
```powershell
phoenix generate --story-file user_stories/login.txt --no-gate
```

---

### `ModuleNotFoundError: No module named 'pages.base_page'`
`pages/base_page.py` is missing. Run:
```powershell
phoenix migrate
```
This adds any missing files to your project. If the error persists, check that `pages/base_page.py` exists in your project root.

---

### Intelligence server won't start / "Connection refused"
- Confirm `ANTHROPIC_API_KEY` is set: `echo $env:ANTHROPIC_API_KEY`
- Check port 8001 is free: `netstat -ano | findstr :8001`
- Start the server: `C:\Phoenix\phoenix-intelligence.exe`

---

### Tests fail with locator or timeout errors
```powershell
phoenix fix
phoenix run --failed-only
```
If the same test keeps failing, watch it in headed mode:
```powershell
phoenix run --headed --slow-mo 1000 tests/<path>
```

---

### Supporting documents not being used
Place the folder next to the story file with the same name:
`user_stories/checkout.txt` → `user_stories/checkout/wireframe.pdf`
